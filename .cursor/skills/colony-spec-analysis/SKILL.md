# Анализ спецификаций

## Быстрый старт

1. Сгенерировать индекс: `python tools/spec_index.py` (из корня репозитория).
2. Открыть `dev/spec_index.md` — таблица файлов, заголовков H1, объёма.
3. Опорные документы: `spec/COLONY_CONQUEST_MASTER_SPECIFICATION.md` (раздел подсистем и интеграции), `spec/technical_architecture_specification.md` (ECS, сеть, стек), `spec/README.md` (порядок чтения), при спорных трактовках — `docs/decisions/` (ADR).

## Что проверять

| Вопрос | Действие |
|--------|----------|
| Все ли домены из мастер-оглавления имеют файл? | Сверить список в мастер-документе с строками в `dev/spec_index.md`. |
| Дубли концепций? | Искать одинаковые термины в разных файлах; выносить канон в одно место или явную ссылку. |
| Интеграционные границы? | Экономика ↔ военные ↔ поселенцы: кто пишет в мировое состояние, очереди сообщений ECS. |
| Трассируемость | Для новой фичи: минимум одна ссылка из мастер-дока или явный подпункт «Связанные спеки»; архитектурные выборы — ADR + карточка канбана. |

## Формат отчёта пользователю

Кратко: **найдено / риск / рекомендация**. Таблица расхождений при необходимости. Не переписывать спеки без запроса — только вывод анализа и точечные правки по согласованию.

## Противоречия с кодом

Когда появится код в `game/`, приоритет: **рабочий код + явное решение в спеке**. Если код отстаёт — карточка в канбане `dev/kanban.json` с меткой `spec-code-drift`.
