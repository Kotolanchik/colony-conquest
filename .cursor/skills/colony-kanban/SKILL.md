# Канбан Colony & Conquest

## Где лежит

- **`dev/kanban.json`** — единственный источник истины для агента (схема ниже).
- **`dev/kanban.md`** — человекочитаемая доска; после правок JSON синхронизируй колонки и карточки в MD.

## Схема JSON

Корень: `version` (number), `updated` (ISO date `YYYY-MM-DD`), `columns` (array).

Каждый элемент `columns`: `id`, `title`, `items` (array).

Каждый `item`: `id` (строка, уникальна в пределах файла), `title`, поля:

- `spec_refs` — массив путей к `spec/*.md`. **Обязательно** для задач с дизайном или кодом (исключение: чистый процесс без изменения `spec/`). Подбор: `dev/BACKLOG_SPEC_INDEX.md`.
- `adr_refs` — массив путей к `docs/decisions/*.md`, если задача связана с принятым ADR или его реализацией.
- `task_file` — путь к `dev/tasks/<id>.md` с детализацией (шаблон: `dev/TASK_CARD_TEMPLATE.md`); заполнять при переводе в **ready** / **review**.
- `labels`, `blocked_by`, `notes` — как раньше.

Стандартные `id` колонок: `backlog`, `ready`, `in_progress`, `review`, `done`, `blocked`. Не удаляй колонки без согласования — агенты ориентируются на стабильные id.

## Как двигать карточку

1. Найти `item` по `id` в текущей колонке.
2. Удалить из старой колонки, вставить в целевую (в конец или по приоритету).
3. Установить `updated` на сегодняшнюю дату.
4. Обновить `dev/kanban.md`: те же колонки заголовками `##`, под каждой — список `- **id** — title` и при необходимости подпункты (spec, блокер).

## Бэклог и отслеживаемость

- **Бэклог** — колонка `backlog`: всё, что ещё не готово к старту.
- **Поток** — `ready` → `in_progress` → `review` → `done`; при блокере — `blocked` с `blocked_by`.
- **Детали** — файл `dev/tasks/<id>.md` (из шаблона `dev/TASK_CARD_TEMPLATE.md`), поле `task_file` в JSON.
- **Решения** — новые ADR в `docs/decisions/`; на карточке `adr_refs` для связи задачи с записью.

## Definition of Done (закрытие карточки в `done`)

Минимум перед переносом в **done**:

1. Решение **законченное** в рамках цели карточки: воспроизводимая проверка (шаги в задаче/README/коде), без обязательного «хвоста» без новой карточки. См. `.cursor/rules/full-solutions-engineering.mdc`.
2. Критерии из `dev/tasks/<id>.md` выполнены (или явно согласовано упрощение в `notes` + отдельная задача на остаток).
3. Если менялись спеки — они сохранены в `spec/`, выполнен `python tools/spec_index.py`.
4. Если принято новое архитектурное решение — есть ADR и ссылки с задачи.
5. Если есть код — сборка без ошибок; тесты или зафиксированный tech debt (отдельная карточка).

## Когда создавать новую карточку

- Новая фича или эпик: `id` в стиле `feat-short-name` или `SYS-001`; **всегда** перечислить все релевантные `spec_refs` (мастер + доменная спека + техспека по необходимости).
- Баг или техдолг: префикс `fix-` / `tech-`.
- При переводе в **ready**: создать `dev/tasks/<id>.md`, проставить `task_file`.

## Мини-шаблон записи в kanban.md

```markdown
## In progress
- **feat-ecs-bootstrap** — Поднять пустой Unity DOTS проект в game/Client
  - Spec: spec/technical_architecture_specification.md
```
